# Indexing: what hewtd treats as a document

`INDEX.md` and `REGISTRY.md` — at the root and in every domain — are **generated
artifacts**. `hewtd index`, `hewtd maintain`, and `hewtd integrate` all rebuild
them from the documents on disk. This page is the contract: what gets indexed,
what never does, where the behavior lives, and how to diagnose it when a doc you
expect is missing.

## What gets indexed

Domain folders are walked **recursively**. A document at any depth inside a
domain is indexed.

```
.documentation/
  standards/
    coding-style.md              → standards/INDEX.md  as  coding-style.md
    backend/
      entity-schema-contract.md  → standards/INDEX.md  as  backend/entity-schema-contract.md
                                 → root INDEX.md       as  standards/backend/entity-schema-contract.md
```

Domain indexes link domain-relative; the root index links root-relative. A
document's title comes from its frontmatter `title`; with no frontmatter title,
hewtd derives one from the **file name** (`entity-schema-contract.md` →
"Entity Schema Contract"), never from the folder path.

> **Recursive indexing landed in 2.7.1.** Before that the walk was flat, so
> every document in a subfolder was invisible to the entire system — never
> indexed, never counted, never drift-checked. See
> [issue #12](https://github.com/TheGlitchKing/hit-em-with-the-docs/issues/12).
> The first `index`/`maintain` run after upgrading will add a row for every one
> of those documents. That is correct, and it is a one-time change.

## What never gets indexed

| Excluded | Why |
|---|---|
| `INDEX.md`, `REGISTRY.md` | Generated artifacts, at any depth — not content. |
| `archive/` | Retired docs. Reserved and excluded from every hewtd scan (see [archival](../README.md#archival-process)). |
| `drafts/` | Not ready to publish. |
| `reports/` | Generated output, not source documentation. |
| `_templates/` | Skeletons, not documents. |
| `node_modules/` | Never yours. |
| A nested `.documentation/` | A doc tree vendored inside your doc tree — somebody else's docs. |
| The vault's `facts/`, `incidents/`, `symptoms/` | **See below.** |

Exclusion is by **path segment**, so `features/archive/old.md` and
`features/x/drafts/y.md` are both excluded, at any depth.

### The vault is indexed by its own generators

The knowledge-base vault (`vault.root`, default
`.documentation/knowledge-base/`) has its own indexes — facts, incidents, and
symptoms are generated by `facts-index`, `incidents-index`, and
`symptoms-index`, and they carry their own lifecycle tiers (`fact`,
`incident-narrative`, `incident-facts`).

Projects commonly also register `knowledge-base` as a **domain**. If the generic
domain indexer also swept those subtrees in, every fact and incident narrative
would be indexed a second time and rendered as if it were an ordinary guide.
So `facts/`, `incidents/`, and `symptoms/` under the vault root are withheld
from the generic index.

Everything *else* in the vault folder — a `README.md`, say — is an ordinary
document and is indexed normally. The exclusion is scoped to the three
generator-owned subtrees, not to the whole vault.

## Indexes are generated, never hand-edited

`hewtd index` and `hewtd maintain` **rewrite the root and domain indexes from
disk**. Any row you add by hand is overwritten on the next run.

This matters because of a workaround that was rational before 2.7.1 and is now
obsolete: since nested docs were invisible, people hand-curated root-index rows
pointing into subfolders — and regeneration then *deleted* them, because the
scan could not see the files those rows pointed at. The practical rule became
"curate the index by hand and never run the indexer," which is backwards.

As of 2.7.1 those documents are generated for you. **Delete the hand-curated
rows and let the indexer own the file.** If you find yourself wanting to add a
row by hand, that is a signal the document is being excluded — work the
diagnostic ladder below instead.

## Where this lives in the code

`listDomainDocFiles()` in `src/generators/regenerate.ts` is the single shared
spine. It has three consumers, and a change to it changes all three:

| Consumer | Uses it for |
|---|---|
| `buildDomainEntries` (`generators/regenerate.ts`) | Building the INDEX/REGISTRY rows |
| The `index-drift` audit rule (`core/audit/auditor.ts`) | Detecting docs on disk that `INDEX.md` does not list |
| The `domain remove` orphan count (`core/domains/manage.ts`) | Counting docs that would be orphaned |

That shared spine is what made the flat-walk bug so durable: the drift rule
compared `INDEX.md` against a document list produced by the *same* blind
function, so nested drift was undetectable by construction. **If you change what
counts as a document, change it here.**

## Troubleshooting

### "A document I wrote is missing from the index"

Work down the ladder:

1. **Is it under an excluded segment?** `archive/`, `drafts/`, `reports/`,
   `_templates/`, `node_modules/`, or a nested `.documentation/` — at any depth.
   This is the most common answer.
2. **Is it inside the vault's `facts/`, `incidents/`, or `symptoms/`?** Then it
   belongs to the vault's own indexes, not the domain index. Look there.
3. **Is it named `INDEX.md` or `REGISTRY.md`?** Those are generated artifacts and
   are never treated as content, even in a subfolder.
4. **Is it in a folder that is actually a configured domain?** Only domains are
   walked. A top-level folder that is not a registered domain — built-in or added
   via `hewtd domain add` — is never scanned. Check with `hewtd domain list`.
5. **Did you regenerate?** Run `hewtd index`.

Then let the tool tell you: **`hewtd audit`** reports an `index-drift` issue
naming every document on disk that its domain's `INDEX.md` does not list.

```
standards/INDEX.md :: INDEX.md lists 2 of 50 document(s) in standards/ —
  unlisted: backend/api-design.md, backend/celery-patterns.md, ...
```

### "`hewtd index` deleted rows from my root INDEX.md"

Expected, if those rows were added by hand — the indexes are generated (see
above). Recover with `git checkout -- .documentation/INDEX.md`.

Pre-2.7.1 this happened *wrongly*: rows pointing at nested documents were deleted
because the scan could not see those documents. On 2.7.1+ the nested documents
are generated for you, so the hand-curated rows are no longer needed. If a row
still disappears, the document it points at is being excluded — work the ladder
above.

### "My index grew by hundreds of rows when I upgraded"

Expected, correct, and one-time. Those documents were always supposed to be
indexed; the flat walk simply could not see them. In the repository that
surfaced this, 215 previously-invisible documents appeared on the first run.

Nothing under `archive/`, `drafts/`, `reports/`, or the vault's generated
subtrees is in that diff. Preview it before committing:

```bash
hewtd index --dry-run
```

### "hewtd crashed with a YAML error"

Fixed in 2.7.1. A document with malformed frontmatter used to abort the whole
run with a `gray-matter` stack trace — and because the walk was flat, a broken
*nested* doc could sit there for months without anyone noticing.

hewtd now indexes such a document anyway, using a filename-derived title, and
warns:

```
⚠ Unparseable frontmatter in features/ux-assist/guided-next-step-ctas-design.md
  — indexing it with defaults. Fix the YAML to control how it appears: ...
```

The document stays findable; fix the YAML to control how it is presented. The
usual culprit is an unquoted value containing a `:` — quote it.

### "My facts and incidents were indexed as ordinary guides"

Should be impossible on 2.7.1+. If you see it, `vault.root` in
`.claude/hit-em-with-the-docs.json` does not point at your vault, so hewtd cannot
tell which subtrees the vault's own generators own. Check it:

```bash
cat .claude/hit-em-with-the-docs.json   # vault.root — default: .documentation/knowledge-base/
```
